## Key Issues

### 1. `react-hooks/set-state-in-effect` — 27 errors

`setState` / data-fetching helpers are called directly at the top level of `useEffect` bodies. The linter flags this as synchronous state mutation within an effect. The pattern in use is:

```ts
// ❌ flagged
useEffect(() => {
  loadData();        // calls setState internally
}, [dep]);

// ❌ flagged
useEffect(() => { load(); }, [load]);
```

**Fix:** The rule targets *synchronous, unconditional* `setState` calls. Wrapping in an async IIFE or keeping the fetch as a named `async` function called inside the effect resolves this. If the rule is a false positive for async loaders, a targeted `// eslint-disable-next-line` with a comment is acceptable.

---

### 2. `react-hooks/purity` — 13 errors

Two sub-patterns:

**a) Components created inside render** — sub-components (e.g. inline form components) are defined inside the body of the parent component, causing them to be recreated on every render.

```ts
// ❌ flagged
function ParentPage() {
  function InnerForm() { return <div />; }   // new reference each render
  return <InnerForm />;
}
```

**Fix:** Move inner components to module scope (outside the parent function).

**b) Impure calls during render** — `Date.now()` / `new Date()` called directly inside `.map()` callbacks in JSX.

```tsx
// ❌ flagged (admin/page.tsx:2636)
const age = r.birth_date
  ? Math.floor((Date.now() - new Date(r.birth_date).getTime()) / ...)
  : null;
```

**Fix:** Move date calculations outside JSX, or wrap in `useMemo` / a pure helper function.
